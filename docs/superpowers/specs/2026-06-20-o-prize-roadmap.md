# 计划书：迈向美赛 O 奖级论文的全自动生成（人类审核、AI 主导提思路）

> 状态：战略计划草案，待用户复核。日期：2026-06-20。
> 目标：让 Mag 在与人类讨论后（主要由 AI 提思路、人类审核），全自动产出一篇**有竞争力冲击 MCM/ICM Outstanding Winner（O 奖）**的论文。

---

## 0. 先说清楚：O 奖意味着什么、以及一个诚实的预期

依据 COMAP 公开说明，Outstanding 是"best of the best"，在**建模与解题、分析、沟通**三个维度都处于全部参赛作品的最高水平；**摘要页（Summary Sheet）是全文最重要的一页**——初轮评审只读摘要。它"既是研究，也是一次演示（presentation）"。

来源：
- [2025 MCM/ICM Results — COMAP](https://www.comap.com/blog/blog-2/item/2025-mcm-icm-results)
- [MCM Instructions — contest.comap.com](https://www.contest.comap.com/undergraduate/contests/mcm/instructions.php)
- [MCM-ICM Tips（procedures & tips）](https://www.contest.comap.com/undergraduate/contests/mcm/flyer/MCM-ICM_Tips.pdf)

**诚实的预期管理（重要）：** O 奖是 top ~0.1–1%、且带主观与竞争性的结果，没有任何方法能"保证"拿 O。因此本计划的可证伪目标是**沿奖项阶梯稳定爬升**，把"偶然能出 O"变成"稳定 Finalist/Meritorious、有机会冲 O"：

```
完整且正确（不崩、对题、可编译）   —— 已基本达到
  → 扎实（真实建模+验证+敏感性+好图+好摘要） —— Meritorious 级，本计划主攻
    → 有深度与创意+极强沟通                  —— Finalist/Outstanding 级，本计划的拉伸目标
```

衡量是否进步**不能靠感觉**——见 §5 评测体系（这是整件事成败的关键）。

## 1. O 奖论文的硬要求（评审视角的 rubric）

| 维度 | O 奖要求 | 评审关注 |
|---|---|---|
| 摘要页 | 一页内讲清问题、方法、关键结果、结论与建议，有说服力 | 初轮唯一阅读对象，决定能否进复审 |
| 问题覆盖 | **每个子问题**都被深入解决，不遗漏 | 完整性 |
| 建模 | 方法**贴题、合理、有创意、严谨**；非教科书套用 | 模型适当性与创造性 |
| 数学 | 推导正确、记号清晰、假设有理有据 | 技术正确性 |
| 数据与求解 | 真实数据、真实计算、结果可信 | 科学性 |
| 验证 | 误差分析、与已知/真值对比、交叉验证 | 结果可信度 |
| 敏感性 | 关键参数扰动分析、稳健性结论 | 模型鲁棒性 |
| 图表 | 多张**信息量大、清晰**的图（数据/模型/结果/敏感性） | 沟通 |
| 写作 | 结构清晰、叙述连贯、强假设论证、记号表 | 沟通（O 奖的胜负手） |
| 一致性 | 摘要=正文=模型=代码=结果，自洽 | 可信度 |

## 2. 现状 vs O 奖：诚实差距分析（已定位到代码）

| 能力 | 现状 | 距 O 奖差距 | 代码位置 |
|---|---|---|---|
| 端到端不崩、对题、可编译 | ✅ 已达到 | — | 全链路（已验证） |
| 论文装配/语言/排版 | ✅ 干净（刚重写） | 摘要仍偏模板、需更强 | `section_writer.py`/`writer.py` |
| 建模深度与贴题 | ❌ **诊断→固定 route 目录→套 solver_module** | **最大差距**：常套通用模型（如 TOPSIS）而非为题目设计模型 | `agents/modeling.py`+`core/modeling_intelligence.py`+`core/model_recipes.py` |
| 模型↔代码↔叙述一致 | ❌ 模型章节(TOPSIS) 与 solver 实算(淘汰一致率)不吻合 | 自洽性破裂 | `modeling.py` vs `solver.py` |
| 子问题逐一求解 | ❌ 基本只产一个模型/结果 | O 奖要每问深入 | workflow 单线 |
| 验证/敏感性/误差 | ❌ 占位（"baseline sensitivity file generated"） | 必须真实 | `agents/validation.py` |
| 图表 | ❌ 约 1–2 张 | O 奖通常 8–15 张有效图 | `agents/visualization.py` |
| 摘要页 | ⚠️ 有但偏模板 | O 奖第一胜负手 | `section_writer`(abstract) |
| 自我对标评审 | ❌ 无 | 需"模拟评审+迭代" | 缺失 |
| 数据接入 | ⚠️ 只配 LLM 时搜索弱 | O 奖常需外部真实数据 | `agents/search_data.py` |

**一句话**：现在能稳定产出"完整、正确、可编译、语言干净"的论文（约 Successful Participant→可能 Honorable Mention）；要上 O，**核心是把"套模板的求解"换成"为题目设计并实现的、自洽且经过验证的研究"，再叠加图表、摘要与模拟评审迭代**。

## 3. 核心策略（8 个关键赌注）

1. **子问题驱动的研究循环**：把题目分解为子问题，对**每个子问题**跑「提方法→定模型规格→生成并执行代码→验证→出图→写作」闭环。O 奖论文逐问深入。
2. **模型↔代码↔叙述强一致**：用 LLM **设计一份"模型规格"(model spec：变量、假设、方程、算法、指标)** → 代码严格按 spec 实现 → 论文严格按 spec 叙述。**废弃"从固定 route 目录里挑"**。这一条直接修掉 TOPSIS 错配。
3. **建模质量靠"方案竞标 + 评审"**：对每个子问题生成 2–3 个候选建模方案，用 LLM 评审团按"贴题/创新/严谨/可行"打分，选优或融合；**人类在此 gate 审核方向**。
4. **验证/敏感性/误差成为一等公民**：自动生成参数扫描、误差对比、交叉验证代码，结果进正文与图。
5. **图表富集**：图计划目标"每子问题 2–3 张有效图"（数据画像、模型示意、结果对比、敏感性曲线），统一风格、矢量优先、caption 绑定结论。
6. **摘要页优先**：把一页摘要当最高价值产物，单独"生成→自评→重写"多轮，对标 O 奖摘要风格。
7. **模拟评审自评迭代（mock judging）**：LLM 评审团按 §1 rubric 给草稿打分 + 产出修订清单 → 迭代，直到分数收敛或人类接受；可用历史 O 奖论文做 RAG 锚定标准。
8. **人在环中的审核 gate**：AI 在每个关键节点提方案，人类审核/编辑/改向（见 §6）。

## 4. 分阶段路线图（每阶段可发布、可验证）

> 每阶段沿用现有节奏：TDD + 真题回归 + 提交推 main。

### 阶段 O0：评测地基（先做，否则无法判断进步）
- 建"模拟评审"打分器（LLM 评审团，输出 rubric 各维度分 + 评语）。
- 收集若干历史**真题 + 公开 O 奖论文**作为锚样本（版权材料仅本地，不入库）。
- 校准：打分器对 O / Finalist / Meritorious 样本能拉开差距。
- 产出 `scripts/score_paper.py` + 回归基线。**这是整件事的标尺。**

### 阶段 O1：模型规格化 + 模型↔代码↔叙述一致（最高优先）
- 新增 `ModelSpec`（变量/假设/方程/算法/输入输出/指标）；LLM 为子问题产出 spec。
- `solver_coder` 改为**按 spec 生成代码**；`writer` 的模型章节**按同一 spec 叙述**。
- 废弃/降级"固定 route 目录套用"为最后兜底。
- 验收：模型章节描述 = 代码实现 = 结果，三者自洽（mock-judge 的"一致性"维度提升）。

### 阶段 O2：子问题逐一求解
- 从题意理解产出**子问题清单**；workflow 对每个子问题跑 spec→code→validate→figure。
- 论文按子问题组织"模型—结果"。
- 验收：每个子问题都有对应模型、结果、图、结论。

### 阶段 O3：真实验证 / 敏感性 / 误差分析
- 生成参数扫描、与真值/基准对比、交叉验证的真实代码与结果。
- 敏感性曲线图 + 稳健性结论写入正文。
- 验收：validation/sensitivity 不再是占位，有真实数字与图。

### 阶段 O4：图表富集与统一风格
- 图计划目标每子问题 2–3 张；统一配色/字体/矢量；caption 绑定结论与证据。
- 验收：真题产出 8+ 张有效图，渲染清晰。

### 阶段 O5：摘要页引擎
- 摘要"生成→自评(对标 O 奖摘要)→重写"多轮；包含问题、方法、关键量化结果、建议。
- 验收：mock-judge 摘要维度显著高于模板基线。

### 阶段 O6：模拟评审 + 修订循环
- 草稿 → 评审团打分 + 修订清单 → 按清单局部重跑/改写 → 复评，直到收敛或人类接受。
- 验收：同一题多轮迭代后总分单调上升并收敛。

### 阶段 O7：人在环中的审核 gate（见 §6）
- 在方向、模型 spec、子结果、草稿处暂停，AI 给方案，人类批/改/驳。
- 验收：用户能在每个 gate 用最小操作把控走向。

### 阶段 O8：端到端 O 级试跑与评测
- 在多道**历史真题**上跑全流程，用 O0 打分器 + 人工对比历史 O 奖论文。
- 产出差距报告 → 回到 O1–O6 迭代。

**串行建议**：O0 → O1 → O2 → O3 → O5 → O6 →（O4 可并行）→ O7 → O8。

## 5. 如何衡量"是否在逼近 O 奖"（成败关键）

不可证伪的目标等于没目标。三层评测：
1. **自动 rubric 打分器**（O0）：每次产出按 §1 九维度打分，留历史曲线。
2. **历史真题对照**：在已知 O 奖论文的历史题上跑，和真 O 奖论文逐维度对比（人工 + 打分器）。
3. **人工专家抽审**：关键里程碑请人（你）按真实评委视角盲评。

子指标随阶段跟踪：子问题覆盖率、模型-代码-叙述一致性、是否有真实敏感性、有效图数量、摘要质量分、编译通过率。

## 6. 人在环中（AI 提思路、人类审核）的 gate 设计

| Gate | AI 提供 | 人类操作 |
|---|---|---|
| G1 题意 + 子问题分解 | 子问题清单 + 风险 | 批准/增删/改 |
| G2 每子问题建模方向 | 2–3 候选方案 + 评审打分 + 推荐 | 选/改/驳回重提 |
| G3 数据方案 | 需要哪些数据、可得性、代理方案 | 批准/上传/换思路 |
| G4 子问题结果 | 结果 + 验证 + 图 | 接受/要求重跑 |
| G5 草稿审稿 | 全文 + mock-judge 评分与修订清单 | 接受/指定修订 |
| G6 终稿 | 提交包 | 放行 |

实现上：每个 gate 暂停并落盘"待审提案"，人类用 slash 命令或自然语言批/改；默认低风险项（重画图、小改文案）可设自动通过。

## 7. 风险与诚实caveat

| 风险 | 说明 / 应对 |
|---|---|
| O 奖主观 + 竞争（top ~0.1–1%） | 不承诺必得 O；目标是稳定爬阶梯，O 作拉伸 |
| LLM 建模创造力/数学正确性上限 | 方案竞标+评审+验证+人审多重过滤；关键数学可加符号/数值校验 |
| 评测本身就难（评审主观） | O0 打分器需用真实样本校准，并以人工盲评兜底 |
| 成本与时长（多子问题×多迭代） | 阶段化、可配预算、缓存中间产物 |
| 版权（历史题/ O 奖论文） | 仅本地用于校准/RAG，不入库 |
| 过拟合到打分器 | 打分器 + 人工双轨，定期换样本 |

## 8. 落地起点（建议立即可做的前两步）

1. **O0 评测地基**：先有标尺，否则后续一切"改进"不可验证。
2. **O1 模型规格化 + 一致性**：当前最大、最值钱的差距（直接修掉"套 TOPSIS、模型≠代码"），且能立刻被 O0 打分器量化。
